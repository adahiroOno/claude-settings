# コスト最適化の設計根拠

Claude Code のトークン消費が大きくなるとき、原因はほぼ次の4つに分解できる。

```
コスト = リクエスト数 × ( 実効入力トークン × 入力単価 + 出力トークン × 出力単価 )

実効入力トークン = 非キャッシュ入力 × 1.0 + キャッシュ書込 × 1.25 + キャッシュ読出 × 0.1
```

つまりレバーは「**単価(モデル)**」「**入力量(コンテキスト)**」「**キャッシュ係数**」「**出力量**」の4つ。本リポジトリの設定はこの4つを個別に押さえる。
従量課金ではそのままコストに、サブスクリプションではレートリミット(利用枠)の消費に効く — 施策は共通。

## 0. 価格の前提(参考: Anthropic API 定価、/1M トークン)

| モデル | 入力 | 出力 | 位置づけ |
|---|---|---|---|
| Opus | $5.00 | $25.00 | 難タスク時のみ手動切替 |
| Sonnet | $3.00 | $15.00 | **デフォルト** |
| Haiku | $1.00 | $5.00 | バックグラウンド・探索サブエージェント |

重要なのは絶対額より**比率**: Opus→Sonnet で約 40% 減、探索を Haiku に逃がすとその部分は 1/3〜1/5。出力単価は入力の 5 倍 — 「出力を削る施策」は見た目以上に効く。

### 設定競合の解決ルール(既存資産との衝突時に何が起きるか)

導入した設定が既存のスキル・ルール・フックと競合したとき、**解決のされ方は面ごとに異なる**。
「勝敗が決まる」「累積される」「モデルの解釈に委ねられる」の3種を区別することが重要:

| 競合する面 | 解決のされ方 | 具体的な挙動 |
|---|---|---|
| settings のスカラ値(model, env 変数, alwaysThinkingEnabled) | **勝敗**(優先順位) | managed > CLI > local > project > user。高優先が上書き。install.sh の保持マージは同名キーをテンプレート優先 |
| permissions(allow / deny) | **勝敗**(deny 優先) | deny が allow に勝つ(安全側)。過剰ブロックが起きたら deny を緩める。install.sh は新旧を和集合 |
| hooks(同一イベントに複数) | **累積**(全部走る) | 全フックが順に実行。**1つでもブロックすれば止まる**(最も制限的なものが勝つ)。install.sh は新旧を和集合でマージし既存フックを保持 |
| CLAUDE.md(グローバル+プロジェクト) | **モデルの解釈**(非決定的) | 連結され、矛盾はモデルが判断。出力がぶれる原因になるため矛盾は解消すべき |
| 出力スタイル | **排他**(1つのみ) | 同時に1つしか有効にならない。切替で前のスタイルは無効 |
| スキル description | **累積 → 誤発火リスク** | 全 description が常駐。発火条件が曖昧・重複だと無関係タスクで発火(F-4) |

監査では I 節(設定の競合)でこれらを点検する。特に **hooks の累積**は、他フックがモデル呼び出しを
していると予算ガードのコスト計算の外側で消費が起きる(G-3)ため要注意。

### モデル・effort の自動ハンドリングの限界(仕様の境界)

「タスクに応じて model / effort を決定論的に自動切替」はフックではできない。フック出力の仕様は
「ツール実行のブロック/許可」と「コンテキスト注入」だけで、リクエストの model / effort を
書き換える制御ポイントがないため。自動化できるのは次の2種類まで:

- **静的な環境設定**: `model`(既定 Sonnet)、`alwaysThinkingEnabled: false`(全ターン思考しない)。起動時に効く。
- **役割ベースの割り当て**: サブエージェント単位のモデル固定(例: explore = haiku)。重い定型タスクが自動で安いモデルに回る。**新しい軽量・独立タスクが出てきたら、専用サブエージェント化して haiku を割り当てるのが正攻法。**

main モデルの動的切替と effort の動的調整は、フックからは不可能で、モデル/ユーザーの判断による
半自動が限界(CLAUDE.md のモデル規範がこれを担う)。特に effort は Claude Code ハーネスが
設定するパラメータで、モデルもフックも実行時に**変更**はできない点に注意(現在の effort レベルの
**表示**は statusline で可能 — 公式 `effort.level` フィールドを使用)。判定自体にモデルを使う
「自動トリアージ」はトークンを浪費する反パターンなので採らない。

## 1. モデルミックス

- **メイン = Sonnet**(settings.json の `"model": "sonnet"`)。コーディングの大半は Sonnet で品質が足りる。難所だけ `/model` で Opus に一時切替し、終わったら戻す。
- **探索サブエージェント = Haiku**(`agents/explore.md` の `model: haiku`)。探索はトークン量が多く品質要求が低い — モデルミックスの効果が最大化される場所。

## 2. プロンプトキャッシュを壊さない

Claude Code はプロンプトキャッシュを自動で使う。ヒットすれば入力の大半が 0.1 倍になるため、**キャッシュヒット率は実質的に最大のコスト変数**。

キャッシュは「プレフィックスの完全一致」。先頭側 = システムプロンプト・ツール定義・CLAUDE.md・会話履歴の順に積まれ、途中の1バイトの変化でそれ以降が全部無効になる。

壊すもの(監査で検出する):
- **フックや CLAUDE.md への動的文字列の注入**(日時、`git status` 出力など) — 毎ターン先頭から無効化される最悪パターン
- **セッション途中の MCP サーバー追加/削除・モデル切替** — ツール定義は最先頭にあるため全滅
- **キャッシュ無効化系の環境変数の残骸**
- **5分超の放置**(TTL切れ) — 再開時に 1.25 倍の書き込みからやり直し

## 3. コンテキスト最小化

入力トークンには2種類ある: **固定費**(毎リクエスト必ず乗る: CLAUDE.md、MCP ツール定義)と**変動費**(会話に積まれたファイル内容・ツール結果)。

### 固定費を削る
- **CLAUDE.md は 2,000 トークン以下**(グローバル+プロジェクト合計の目安)。書くのは「毎セッション必要な事実」だけ。手順書・設計知識は**スキルへ**(スキルは description 1行だけが常駐し、本文は必要時のみ読み込まれる = progressive disclosure)。@import は import 先も毎回読み込まれるため常用しない。
- **MCP サーバーは使うものだけ**。接続中の全ツール定義が毎リクエストの入力になる。プロジェクト単位の `.mcp.json` + `enableAllProjectMcpServers: false` で絞る。

### 変動費を削る
- **読ませない**: `permissions.deny` でロックファイル・生成物・シークレットの Read を遮断。Bash 経由の `cat` は `guard-heavy-read.sh` フックで補完。一度読んだ巨大ファイルは**以降の全ターンでコンテキストに残り続ける**点が重要(1回の事故が尾を引く)。
- **探索を隔離する**: 横断調査は explore サブエージェントへ。中間結果(大量のファイル内容)はサブエージェント側のコンテキストで消費され、メインには結論だけが返る。
- **セッションを切る**: タスクが変わったら `/clear`。前タスクの文脈を引きずると、その分をキャッシュ済みでも毎ターン 0.1 倍で払い続ける。autocompact は有効のまま(溢れてからのやり直しの方が高い)。

## 4. thinking / 出力の制御

- `alwaysThinkingEnabled: false` — 全ターン thinking は簡単なタスクで純粋な無駄(thinking も出力単価)。難タスクではプロンプトで明示的に思考を求める。
- グローバル CLAUDE.md の出力規範(結論先行・変更箇所のみ・全文再掲禁止)は出力トークン(単価5倍)を直接削る。
- `CLAUDE_CODE_MAX_OUTPUT_TOKENS` は原則いじらない。小さすぎる値は途中切断→リトライで逆効果。
- **オプトインの `terse` 出力スタイル**(`output-styles/terse.md`): 説明がほぼ不要な定型作業の連続フェーズだけ `/output-style terse` で地の文を電報調に切り詰める(コード・成果物・エラー説明は通常品質のまま)。終わったら `/output-style default` で戻す。地の文の節約は $0.10/ターン基準の数%規模なので**常用はしない** — 説明を削りすぎて聞き直しが発生すると逆に高くつく。スタイル本文自体も有効中は常駐する(terse は約300トークンに抑えてある)。

## 5. 非本質呼び出しの削減

- `DISABLE_NON_ESSENTIAL_MODEL_CALLS=1` — 装飾的なモデル呼び出しを止める
- `CLAUDE_CODE_DISABLE_NONESSENTIAL_TRAFFIC=1` — 更新チェック等の非本質トラフィックを止める(トークンよりネットワーク・ノイズの削減)

## thinking(判断根拠)のコスト構造と事後アーカイブ

thinking(拡張思考)はモデルの判断根拠に近いテキストだが、**生成された時点で出力トークンとして
課金される**。これは `thinking.display` の設定(`"omitted"` / `"summarized"`)に関わらず発生する
不可避のコストで、display は可視性のみを制御する(summarized にしてもコストは増えない)。

`"omitted"`(既定)では中身が空文字列で返るため、後から中身を取り出すことはできない
(課金は発生しているが内容は不可視)。判断根拠を残したい場合は `"summarized"` への切り替えが必要。

一度生成された thinking を**トークンを使わずに**アーカイブする方法: Claude Code は各セッションの
トランスクリプトを `~/.claude/projects/*/*.jsonl` に自動保存しており、thinking ブロックも含まれる。
`export_thinking.sh`(jq のみ、モデル呼び出しなし)でこれを Markdown に抽出できる — アーカイブ
作業自体の追加コストはゼロ。ただし抽出できるのは、その時点で "summarized" になっていたセッション分のみ。

## 6. セッション予算ガード(サーキットブレーカー)

「高くなりすぎたセッションを止める」を**モデルの自制ではなくハーネスの強制**で実現する仕組み(`hooks/session-budget-guard.sh`)。

- 毎ツール呼び出し前・毎プロンプト送信前に、セッションのトランスクリプト(各ターンの usage 記録)から推定コストを集計する。**モデル呼び出しを一切しない決定論的スクリプト**なので、ガード自体のコストはゼロ。
- 3段構えで効く:

| 段階 | 条件(既定) | 動作 |
|---|---|---|
| **コンテキスト肥大ガード** | ctx > 12万トークン(`CLAUDE_CTX_LIMIT_TOKENS`) | セッション1回、ブロックして「以後の読み込みは explore へ・handoff 更新・/clear 提案」を指示。膨張をコスト事故になる前に自動検知する |
| **ペースガード** | $0.10/ターン(= **10ターン ≒ $1**)を超過 | 5ターンに1回まで、ツール実行を一度ブロックして効率化を指示。「品質を落とすのではなく無駄を削る」(再読の禁止・explore 委譲・出力の絞り込み・重複検証の排除)を明示し、タスク完遂は最優先のまま |
| **警告閾値** | セッション累計 $2.5 | 一度だけブロックし、残作業の絞り込み・/compact 提案を注入 |
| **上限** | セッション累計 $5 | 以降の全ツール実行と新規プロンプトをブロック。Claude は要約して終了し、ユーザーには /clear が案内される |

介入メッセージの届き先はイベントで異なる(公式フック仕様): **PreToolUse の exit 2 は stderr がモデルに渡る**ため是正指示として機能する。**UserPromptSubmit の exit 2 はプロンプトを消去し stderr はユーザーにのみ表示される**ため、モデルへの指示には使えない — だからペース・肥大・警告の3段は PreToolUse でのみ発火させ、UserPromptSubmit では上限超過の最終ブロック(ユーザーへの案内)だけを行う設計にしている。

### 自動化の境界(ユーザーが何もしなくてよい範囲)

「ユーザーが /clear 等を意識しなくてもコストが増えない」は、Claude Code の仕様上ここまで自動化できる:

- **完全自動(ユーザー操作ゼロ)**: 予防層(deny・読み込みガード・explore 委譲・キャッシュ保全)、ペース/コンテキストの監視と介入、上限での強制停止。放置してもセッション上限額を超えることはない。
- **1操作だけ残る**: コンテキストを実際に縮める `/clear` は、仕様上フックやスクリプトからは実行できない(組み込みの autocompact は自動だが、コンテキスト上限間際でしか発火せずコスト的には手遅れ)。そこで ctx ガードが「タイミングの判定・handoff の保存・提案文の用意」まで自動で済ませ、**ユーザーの仕事は提案されたときに /clear と打つだけ**(引き継ぎ済みなので無損失)に縮めてある。
- 提案を無視して続けた場合でも、ペースガード→警告→上限の各段が順に効くため、コストが青天井になることはない。

- 調整は `env` の `CLAUDE_TURN_BUDGET_USD` / `CLAUDE_SESSION_BUDGET_WARN_USD` / `CLAUDE_SESSION_BUDGET_USD`。
- ステータスラインが**閾値に対する使用率をバーと絵文字で可視化**する(Claude Code 公式の statusline JSON を利用)。表示例:
  ```
  🤖 Opus 4.8 │ ⚡ high │ 💰 $2.50/$5 █████░░░░░ 50% │ 🧠 60k/120k █████░░░░░ 50% │ 🎯 10T:$0.65 │ 🔄 7 │ 💾 90% │ 📊 5h:24% 7d:41% │ ✍️ terse │ 📝 +12/-3
  ```
  - **⚡ effort**: 推論努力レベル(low〜max。公式 `effort.level`。モデル対応時のみ)
  - **💰 予算バー**: セッションコスト vs 上限($5)。使用率バー(█塗り/░空)と実値%。80%で ⚠️、超過で 🛑
  - **🧠 ctxバー**: 現在のコンテキスト vs 閾値(12万)。公式 `context_window.current_usage` 由来。90%超・200k超で 🔴
  - **🎯/🔥 ペース**、**💾 キャッシュ率**(50%未満で赤)、**🔄 ターン数**、**📊 レート制限**(サブスク時)、**✍️ スタイル**、**💭 thinking**(有効時)
  - **🎫 トークン内訳**(`CLAUDE_STATUSLINE_TOKENS=1` で有効化): 直近リクエストの新規/読出/書込/出力。既定オフ(横長回避)
  - 全項目の凡例は README「ステータスラインの見方」を参照。閾値は env に連動

ペースガードが「品質を保ったまま」成立する理由: 目標ペースの達成手段を**効率化(キャッシュ活用・委譲・重複排除)に限定**し、モデルのダウングレードや検証の省略を指示しない。ペース超過の主因は通常、品質に寄与しない消費(同じファイルの再読、メイン文脈での探索、全文再掲)であり、そこだけを削らせる。

なぜ「エージェントに見直させる」方式にしないか: LLM ベースの監視はそれ自体がトークンを消費し、実行・遵守の保証もない。**強制は決定論的フック、見直しの頭脳は /cost-audit** という分担が正しい。

この仕組みの限界(誠実に):
1. **進行中の1リクエストは止められない** — 超過の検知は次のフック発火時点。上限には最大1リクエスト分の遊びがある。
2. **推定値である** — 単価は代表値(モデル名から推定)。請求額と数%の誤差は出る。余裕を持った閾値にする。
3. **クライアント側の自衛である** — 設定を外せば無効になる。組織として絶対の上限が必要なら、請求側(プロバイダのコンソール/管理者)での支出上限・クォータ設定を併用する。

### ガードの弊害を出さないための設計

強制ブロックには副作用がありうる。次の対策を組み込んである:

| 想定される弊害 | 対策 |
|---|---|
| 上限ブロックで作業状態ごと失われる | **グレースレーン**: 上限超過中も `.claude/handoff.md` への Write/Edit だけは許可。ブロックメッセージが「①引き継ぎメモを書く → ②要約 → ③再開案内」の終了手順を指示する |
| ペース警告が品質低下を招く | 是正手段を「無駄の削減」に限定し、「タスクの完遂が最優先」を明示。警告は5ターンに1回まで |
| 短いセッションでの誤発火 | ペース判定は3ターン目から。許容額に+1ターン分の余裕 |
| フック自体の不具合で作業不能になる | **フェイルオープン**: jq 不在・トランスクリプト不在・解析失敗時はすべて素通し(exit 0) |
| ガード自体が長時間セッションで重くなる | **差分解析**: 処理済みバイト位置と累計を状態ファイルに保持し、追記分だけを解析(O(増分)。書きかけ行は次回に繰り越し、状態破損時はフル再計算) |
| statusline の解析が重くなる | ターン数はガードの状態ファイルを再利用、ctx は末尾100行のみ解析(全体を読まない) |
| /tmp に警告マーカーが溜まる | SessionStart フックが2日超のマーカーを自動掃除 |
| Claude Code 本体の更新で前提が変わり、フェイルオープンなガードが**静かに無効化**される | **三重の仕様ドリフト検知**: ①ランタイム(読めない入力・進行中なのに集計ゼロを1回だけ警告)②カナリア自己診断 `selftest_guard.sh`(既知データ一致・ブロック作動・実トランスクリプト解釈)③SessionStart の本体更新検知→再監査提案。新設定キー等の追加型変化は監査フェーズ0の「未分類」で捕捉 |

### /clear 後の作業再開 — ハンドオフ方式(常駐メモリを使わない理由)

「/clear すると文脈が消えて作業が再開できない」問題への標準的な解は CLAUDE.md への自動メモ追記(常駐メモリ)だが、これは**毎リクエストの固定費**になり本末転倒。代わりに**ハンドオフ方式**を採る:

1. **保存**: 作業の中断・終了時(予算警告・上限時を含む)に `.claude/handoff.md` へ 目的/完了済み/未完了と次の一手/重要な決定 を **40行以内**で書く(グローバル CLAUDE.md の規範 + ガードの指示)。
2. **検知**: SessionStart フック(`handoff-notice.sh`)が、48時間以内の handoff がある場合のみ「存在を知らせる3行」をコンテキストに注入する。
3. **再開**: ユーザーの依頼が続きの作業のときだけ Claude が本文を Read して再開。完了したらファイルを削除する(陳腐化メモの誤再開を防ぐ)。

トークンコストの比較(10ターン≒$1 基準への影響):

| 方式 | コスト構造 | 10ターンあたりの負担 |
|---|---|---|
| 常駐メモリ(CLAUDE.md 自動追記) | メモ全文 × **毎リクエスト**(キャッシュ 0.1 倍でも累積、肥大化も進む) | メモ1,000トークンなら数千トークン規模 = 数%〜十数% |
| **ハンドオフ方式** | 通知 約110トークン × セッション1回(存在時のみ)+ 本文 約500トークン × 再開時1回 | **1% 未満** |

handoff.md は一時ファイル。project-template に同梱の `.claude/.gitignore` が handoff.md と settings.local.json を除外する。

## 手戻りの経済(プロンプト品質がコストレバーである理由)

曖昧な依頼は、モデルの設定をどれだけ最適化しても消せないコストを生む:

1. **投機的探索** — 意図を推測するために不要なファイルまで読む(数千〜数万トークンの入力)
2. **やり直し** — 誤解に基づく実装を捨てて作り直す(探索+出力の二重払い)
3. **残骸の再送** — 誤った文脈(読んだファイル・捨てた実装)はコンテキストに残り、**以降の全ターンで入力トークンとして再送され続ける**

対して、実行前の確認・整形のコストは1往復数百トークン。桁が2つ違うため、大きめの作業では「聞いてから動く」が常にコスト優位になる。本リポジトリでは3つの仕組みで実装している:

| 仕組み | 種別 | 動作 |
|---|---|---|
| CLAUDE.md「依頼の受け方」規範 | 自動(誘導層) | 大きな作業の依頼が曖昧なとき、探索前に確認質問を1つだけしてから着手 |
| `/refine` スキル | 半自動(ユーザー起動) | 依頼を 目的/範囲/制約/完了条件/想定外 の仕様に整形し、不明点のみ質問(最大3問・1往復)してから実行。整形段階ではファイル探索をしない |
| handoff パターン | 自動(SessionStart 通知) | /clear 後の再開時に前回の仕様・決定を引き継ぎ、再説明・再探索の手戻りを防ぐ |

なお `/refine` の出力形式で最重要なのは**完了条件**。完了条件が書けない依頼は範囲が発散しており、着手すればほぼ確実に手戻りが発生する。

## 7. 計測 → 監査のループ

削減は一度きりでは維持できない。CLAUDE.md の肥大化と MCP の増殖は放っておくと必ず起きる。

- **セッション中**: ステータスライン(モデル・累計コスト)と `/cost`
- **週次**: `/cost-audit` — 設定ドリフトの検査(チェックリスト全項目)
- **チーム運用**: OTEL テレメトリ(`CLAUDE_CODE_ENABLE_TELEMETRY=1` + OTLP エクスポート)で `claude_code.token.usage` を type 別(input / output / cacheRead / cacheCreation)に集計する。**cacheRead の比率がキャッシュ健全性の KPI**。モデル別ディメンションで高価なモデルの使用率も監視できる

## アンチパターン早見表

| アンチパターン | なぜダメか | 代替 |
|---|---|---|
| 全部 Opus | 単価 1.7 倍を常時払う | Sonnet 基本 + 手動切替 |
| CLAUDE.md に手順書・設計書を全部書く | 毎リクエストの固定費 | スキル化(オンデマンド) |
| MCP サーバーを全プロジェクトでグローバル接続 | ツール定義が毎回入力に | プロジェクト単位で有効化 |
| フックで日時や git 状態を毎ターン注入 | キャッシュ全滅 | 注入しない |
| 巨大ファイルを気軽に Read | 以降の全ターンに残留 | deny + rg/jq で部分抽出 |
| 1セッションで何でもやる | 無関係文脈を払い続ける | タスク毎に /clear |
| コストを測らずに設定だけ変える | 効いたか分からない | statusline + /cost + /cost-audit |
| 曖昧なまま大きな作業を依頼 | 投機的探索+やり直し+残骸の再送 | /refine で仕様化(または確認1問を待つ) |
