# refine — 実行前プロンプト整形

依頼をすぐ実行せず、まず仕様として固める。曖昧なまま着手した場合の投機的探索・手戻り
(数千〜数万トークン+誤った文脈の残留)を、数百トークンの整形1回で防ぐのが目的。

## 手順

1. **判定**: 依頼が次のいずれかなら「整形不要、このまま実行しますか?」と提案する
   (軽微な作業は整形のオーバーヘッドの方が高い):
   - 単一ファイルの軽微な修正 / 事実の質問 / 定型作業
2. **整形**: 依頼を下の型に再構成する。**意図が読み取れない項目は推測で埋めず「要確認」に残す**。
   整形の材料集めにファイル探索をしない(この段階ではトークンを使わない)。
3. **確認**: 整形結果を提示し、「要確認」は選択肢つきで質問する(最大3つ、1往復で確定させる)。
4. **実行**: 承認された仕様に沿って実行する。仕様にない作業を追加しない。
   完了時は「完了条件」に対する結果を報告する。

## 出力形式

```
📋 整形済み依頼
目的:     なぜやるか / 何のためか
範囲:     触ってよいもの / 触らないもの
制約:     技術・互換性・スタイル上の条件
完了条件: 何が確認できたら完了か(検証方法まで)
想定外:   作業中に別の問題を見つけたときの扱い(直す / 報告に留める)

要確認:   (あれば番号つき選択肢で)
```

## 判断基準

- 「完了条件」が書けない依頼は、まだ着手すべきでない依頼。ここが最重要項目。
- 質問は仕様を確定させるものに限る(好みの確認や網羅的なヒアリングはしない)。
